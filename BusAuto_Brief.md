### KPI і зарплата закупівельника

**Принцип:** фіксована частина + бонус за результат. Штраф за дефіцит НЕ застосовуємо — забагато факторів поза контролем.

| Складова | Сума | Умова |
|---|---|---|
| Фіксована частина | 15 000–20 000 ₴/міс | Завжди |
| Бонус за маржу | до 5 000 ₴ | Середня маржа по нових закупівлях > 30% |
| Бонус за оборотність | до 3 000 ₴ | Dead stock (тільки товари після дати старту) < 15% від вартості складу |
| Fill rate | Моніторинг, не штраф | Тільки по товарах з 3+ замовленнями за 90 днів |

**Важливо про dead stock:** новий співробітник НЕ відповідає за висяки що були до його старту. Фіксуємо базовий знімок складу в день виходу, все що до — виключається з KPI.

**Пояснення бонусу за маржу:** закупівельник контролює ціну закупівлі, але не ціну продажу (прайс фіксований). Якщо знайшов постачальника дешевше на 50 ₴ — маржа на цей товар зростає. Бонус нараховується коли середня маржа по його закупівлях > 30%.

**Fill rate:** відсоток рядків замовлень виконаних одразу з наявності. Рахується тільки по товарах з регулярним попитом (3+ замовлення за 90 днів) — тобто тих, які закупівельник МАВ тримати в наявності.

**Орієнтовна зарплата (розрахунок):**
- Ринок Харків, авторинок: 18 000–22 000 ₴ фікс
- Реалістичний бонус за маржу (перший рік): 2 000–3 000 ₴/міс
- Реалістичний бонус за оборотність: 1 000–2 000 ₴/міс

- Максимум при ідеальних KPI: ~28 000 ₴/міс

**Дані для KPI є в дашборді:** `busauto_warehouse_dashboard.html` — dead stock %, ABC-аналіз, регулярний попит (фільтр "🔄 Регулярний попит" — товари з 3+ замовленнями за 90 днів).

## Shafer Photo Import

**Скрипт:** `shafer_photo_import.py` | **Запуск:** `ЗАПУСТИТИ_SHAFER_ФОТО.bat`

### Алгоритм (2 кроки на товар)
1. **GET** `https://market.shafer.ua/?w=base64({"s":"ARTICLE"})` → парсимо HTML, шукаємо `data-template="ARTICLE_({index}).jpg"` → витягуємо `data-prod=NNNNN` (унікальний ID товару на сайті Shafer)
2. **POST** `https://market.shafer.ua/ajax/` з `action=showGallery, id=NNNNN, market=0` (multipart/form-data) → повертає `{"gallery": {"photoId": "/images/fullimage/NNNNN_photoId_ARTICLE(N).jpg", ...}}`

### Логіка збереження в Odoo
- **Перше фото** з галереї → замінює `image_1920` (головне фото `product.template`)
- **Решта фото** → зберігаються як `product.image` (додаткові)
- Старі доп. фото очищаються перед записом (немає дублів)
- Файли що не є JPEG/PNG/GIF/WebP — пропускаються

### Важлива технічна інформація
- **Авторизація:** cookies з браузера → `shafer_cookies.json` (простий dict `{name: value}`, НЕ Playwright формат)
  - Потрібні куки: `PHPSESSID`, `uid`, `shaferOnlyNew`
  - Session check: GET `/` і перевіряємо наявність `o-user-cart` в HTML (не через `/ajax/`)
- **`/ajax/` повертає HTML** для більшості запитів — це нормально! Тільки `showGallery` повертає JSON
- **Фото розміщені на:** `https://globalavto.com.ua/images/fullimage/` (головні фото запчастин) і `https://market.shafer.ua/img/review/technotes/` (технічні рисунки)
- **Brand ID Shafer в Odoo:** 771
- **Прогрес:** `shafer_processed.txt` — список оброблених `product.template.id`
- **Режими:** `[T]` тест одного артикулу, `[R]` скинути прогрес, `[Enter]` повний запуск
- **Куки expire:** якщо каже "куки протерміновані" → відкрити Chrome → market.shafer.ua → F12 → Console → `document.cookie` → скопіювати PHPSESSID/uid/shaferOnlyNew → оновити `shafer_cookies.json`

## B2B/B2C — архітектурне рішення на перспективу (фаза 2, після фіксу кошика/реклами)

**Питання:** окремий сайт для опту (B2B) і роздробу (B2C), чи один?

**Рішення: один домен/Odoo-бекенд, не два сайти.** Два домени подвоюють SEO-канібалізацію (і так 774k непроіндексованих сторінок через дублі), синхронізацію прайсів, кошики, рекламу, підтримку — суперечить меті "вийти з оперативки".

**Цільова архітектура:**
1. **Окремий пошуковий індекс для каталогу 15 млн SKU + кроси-аналоги** — НЕ в Odoo product.template (15 млн записів покладе систему, у нас вже MemoryError на 120к). Окрема база: PostgreSQL з повнотекстовим пошуком (простіше підтримувати, можна на тому ж сервері) або Elasticsearch (якщо потрібен нечіткий пошук/фасети). Рекомендація — почати з PostgreSQL через відсутність in-house розробника.
2. **ETL для прайсів постачальників** (email/API, різні валюти) → парсинг → нормалізація валюти → матчинг по артикулах/кросах з індексом → НЕ пишеться напряму в Odoo.
3. **Odoo залишається "істиною"**: товари в наявності + ціни + замовлення + склад + бухгалтерія. Тільки опубліковані товари потрапляють у product.template.
4. **B2C-вітрина** (busauto.ua) — публічний каталог, замовлення → Odoo SO як зараз.
5. **B2B-портал** — не окремий домен, а піддомен/розділ (напр. `opt.busauto.ua` або `/b2b/`) з логіном: персональні прайслисти (Odoo pricelist per partner), масове замовлення по артикулах (CSV/список), пошук по всій базі 15 млн через індекс. Замовлення теж падають в Odoo, інший pricelist/source.
6. Якщо товару немає в Odoo, але є в індексі — сайт пропонує "замовити під клієнта", при підтвердженні скрипт створює товар+SO в Odoo через API.

**Статус:** ідея на майбутнє, не блокує поточні задачі (кошик busauto.ua — основний залишок перед запуском реклами).
